> **Purpose:** This document gives an AI agent (Claude, Gemini, etc.) full context on the LEMNI backend project — what it does, how it's structured, what's been built, what conventions to follow, and what remains to be done.

## 1. What Is LEMNI?

LEMNI is a **subscription billing infrastructure platform** for African merchants. It enables merchants to:
- Accept one-time payments and recurring subscriptions via Nomba (Nigerian payment gateway)
- Share payment links publicly (no-code, no SDK needed)
- Manage billing plans, customer records, and subscriptions via a REST API
- Receive real-time payment event webhooks from Nomba
- Get Telegram alerts for billing events

**Key Distinction:** LEMNI does **not** build its own card form. Nomba provides the hosted checkout UI. LEMNI orchestrates sessions, manages state transitions, and handles async billing workflows.

## 4. Database Schema

All tables are in `src/database/schema.ts`. **Never add columns outside of this file.**

```
merchants          — id, name, email (unique), hashed_password, webhook_url,
                     telegram_chat_id, default_redirect_url, created_at
api_keys           — id (keyId), merchant_id, hashed_key, environment, is_active, created_at
customers          — id, merchant_id, email, nomba_token, metadata (JSON), created_at
plans              — id, merchant_id, name, amount, billing_model, interval,
                     trial_days, trial_require_card, grace_period_days, created_at
subscriptions      — id, customer_id, plan_id, status, current_period_end,
                     trial_end, created_at
                     status enum: trialing | active | past_due | canceled
transactions       — id, subscription_id, amount, status, nomba_ref,
                     payload (JSON), response (JSON), created_at
                     status enum: pending | success | failed
dlq_jobs           — id (BullMQ job ID), subscription_id, payload (JSON),
                     error_reason, retry_history (JSON array), failed_at
```

**Migration protocol:** Always use CLI — never write SQL manually.
```bash
pnpm drizzle-kit generate   # detects schema changes → generates SQL
pnpm drizzle-kit migrate    # applies pending migrations to DB
```

## 5. Authentication Model (Two Auth Methods)

### 5A. Merchant Dashboard — JWT Auth
Used by: merchant dashboard frontend, admin endpoints.

```
POST /auth/signup              → { id, email, name }
POST /auth/login               → { accessToken, refreshToken, merchant }
POST /auth/refresh             → { accessToken }
POST /auth/logout              → { message }
POST /auth/reset-password      → { success }
```
- JWT payload: `{ sub: merchantId, email }`
- Access token: 1h expiry. Refresh token: 7d expiry.
- Guarded by: `JwtAuthGuard` (`@UseGuards(JwtAuthGuard)`)
- Extract `merchantId` from request: `(req.user as Record<string, unknown>).sub as string`

### 5B. Developer API — API Key Auth
Used by: developers integrating LEMNI checkout into their apps.

```
POST  /api-keys                → { rawKey, keyId, message }   [JWT required]
GET   /api-keys                → [{ id, environment, isActive, createdAt }]
DELETE /api-keys/:id           → { success, message }
```
- Key format: `sk_test_<keyId>_<secret>` or `sk_live_<keyId>_<secret>`
- Only hashed secret is stored in DB (bcrypt). Raw key shown **once** on creation.
- Guarded by: `ApiKeyGuard` on checkout/billing developer routes.

## 6. Checkout Flows

### Flow A: Developer API Flow (API Key required)

```
Developer Backend → POST /api/v1/pay (one-time)
                 OR POST /api/v1/subscribe (recurring)
                    Body: { email, amount|planId, callbackUrl? }

Lemni → creates Customer + Transaction/Subscription records
      → calls Nomba POST /v1/checkout/order
      → returns { sessionId, checkoutUrl }

Developer → redirects customer browser to checkoutUrl (Nomba-hosted form)

Customer fills card on Nomba form
Nomba → POST /api/v1/webhooks/nomba (payment_success | payment_failed)
Lemni → updates Transaction + Subscription status

Developer polls → GET /api/v1/sessions/:sessionId/status
```

### Flow B: Public Plan Link (No auth needed — merchant shares a URL)

```
Merchant creates plan → shares link: https://app.com/checkout?plan=plan-abc

Customer lands on page → sees email input form (merchant frontend)
Customer enters email → merchant frontend calls:
  POST /api/v1/checkout/plans/:planId/sessions
  Body: { email, callbackUrl? }

Lemni → creates Customer + Subscription + Transaction
      → calls Nomba → returns { sessionId, subscriptionId, checkoutUrl }

Merchant frontend → redirects customer to checkoutUrl
(same webhook + status polling as Flow A)
```

## 7. Async Billing Architecture

### Queue Architecture
- **ChargeQueue** (`charge`) — BillingWorker pushes due subscriptions every hour
- **DunningQueue** (`dunning`) — failed charges retry with exponential backoff

### Worker Services

**BillingWorkerService** (`@Cron(EVERY_HOUR)`)
- Queries `subscriptions WHERE status='active' AND current_period_end <= NOW()`
- Queries `subscriptions WHERE status='trialing' AND trial_end <= NOW()`
- For active due: enqueues `ChargeJobPayload` to ChargeQueue
- For expired trials with card (`nombaToken` present): promotes to `active`, enqueues charge
- For expired trials without card: sets `past_due`, logs `trial.ended_no_card`
- Uses cursor-based pagination (PAGE_SIZE=100) to avoid memory spikes

**DunningWorkerService** (BullMQ Worker, concurrency=5 charge / 3 dunning)
- Processes ChargeQueue: calls `nombaClient.chargeTokenizedCard()`
- On success: advances `current_period_end`, sets status `active`
- On 5xx/timeout (retryable): re-enqueues to DunningQueue with backoff
- On 4xx (non-retryable): checks `grace_period_days`; sends to DunningQueue or DLQ
- On circuit breaker OPEN: re-queues to DunningQueue, defers processing
- Stale job detection: discards jobs where subscription is no longer `past_due`
- Exponential backoff delays: `[0ms, 5min, 30min, 2hr]`

**HealthCronService** (`@Cron('*/5 * * * *')`)
- Checks circuit breaker state
- Pauses ChargeQueue if breaker is OPEN
- Resumes ChargeQueue if breaker recovers

**DLQ Flow**
- Exhausted retries → `dlq_jobs` row inserted in Turso
- `GET /admin/dlq` → lists all DLQ entries (JWT protected)
- `POST /admin/dlq/:jobId/replay` → re-enqueues to ChargeQueue, removes DLQ row

**Circuit Breaker** (in-memory, `CircuitBreakerService`)
- States: `CLOSED` → `OPEN` (after 3 failures) → `HALF_OPEN` (after 30s) → `CLOSED`
- Failure threshold: 3 consecutive Nomba 5xx/timeouts
- All Nomba calls go through `nombaClient` which calls `circuitBreaker.recordFailure/Success()`

## 8. OpenAPI / Swagger Contract

```bash
# Check for drift between decorators and openapi.yaml
pnpm lint:openapi

# Regenerate and write openapi.yaml
ts-node -r tsconfig-paths/register scripts/check-openapi-drift.ts --write
```

**Critical rules:**
- `docs/openapi/openapi.yaml` is a **generated artifact** — never edit it manually.
- All API documentation must live in controller decorators (`@ApiOperation`, `@ApiBody`, `@ApiResponse`, `@ApiTags`).
- Use **class-level** `@ApiTags()` to prevent tag duplication.
- Tag hierarchy: `merchant-dashboard/auth`, `merchant-dashboard/api-keys`, `merchant-dashboard/billing`, `developer-apis/checkout`, `merchant-dashboard/dlq`
- After changing any controller decorator, always run `pnpm lint:openapi --write` and commit the updated `openapi.yaml`.
- When any agent modifies or adds an endpoint, it MUST delegate OpenAPI reconciliation to the `openapi-reconciler` agent.

## 9. Validation Convention

Every endpoint uses `ZodValidationPipe` from `src/common/pipes/zod-validation.pipe.ts`.

**Pattern:**
```typescript
// dto file
export const MySchema = z.object({ email: z.string().email(), amount: z.number().positive() });
export type MyDto = z.infer<typeof MySchema>;

// controller
async myEndpoint(@Body(new ZodValidationPipe(MySchema)) body: MyDto) { ... }
```

Never use `class-validator` / `class-transformer`. All validation is Zod-only.

## 15. Nomba API Integration Notes

- **Checkout order**: `POST /v1/checkout/order` → creates hosted checkout URL. Returns `{ checkoutUrl, orderId }`.
- **Tokenized charge**: `POST /v1/checkout/tokenized-card-payment` → charges a saved card token.
- The `nomba_token` (stored in `customers.nomba_token`) is the card token returned by Nomba after a successful first checkout.
- `NombaClient` wraps all calls with idempotency keys and circuit breaker checks.
- 5xx responses → `circuitBreaker.recordFailure()` → may trip breaker to OPEN.
- 4xx responses → non-retryable → handled by dunning grace period logic.
- Auth: credentials sent per-request via `client-id` / `client-secret` headers (no OAuth session management).
