# RF-005: Customer Card Update Flow (Payment Method Update Without Charge)

**Status:** DRAFT (Planning Phase)

**Date:** 2026-07-04

**Scope:** Allow customers to update their payment method (card) when payment fails, without triggering an immediate charge or requiring subscription reactivation.

---

## 1. Goal

When a customer's payment fails (e.g., card expired, insufficient funds), they need a way to:
1. Update their card details with Nomba (tokenize a new card)
2. Store the new token in the system
3. Retry payment on next billing cycle without manual intervention

**Problem Solved:** Currently, customers have no way to update their card except by:
- Waiting for the dunning retry (if grace period > 0)
- Or having merchant manually reactivate (which requires expired token check per UP-04.11)

**New Flow:** Customers can proactively update their card via a public endpoint that:
- Generates a **tokenization-only checkout** (no amount charged)
- Customer enters new card on Nomba's hosted form
- New card token is stored in `customers.nombaToken`
- Subscription remains in `past_due` state
- Next dunning retry uses the new token

Add under **4. BILLING & DOMAIN LOGIC (BillingModule)** (after 4.6):

**4.7. Customer Payment Method Update**
- Implement `POST /api/v1/public/subscriptions/:id/update-payment-method` (public, no auth)
  - Takes subscription ID as path parameter
  - Returns a tokenization-only checkout URL (no amount, just card capture)
  - Customer completes card entry on Nomba's hosted form
  - Nomba webhooks with tokenized card data
  - System updates `customers.nombaToken` with new card token
  - Subscription stays in current state (remains `past_due` if currently in dunning, or `active` if being proactive)
- Update email notification for `payment_failed` event to include a link to this endpoint
- **Unhappy paths:**
  - Subscription not found or belongs to different merchant → return `404 Not Found`
  - Subscription is already `canceled` → return `409 Conflict` (suggest reactivation endpoint)
  - Tokenization checkout generation fails (Nomba 4xx) → return `502 Bad Gateway`
  - Customer doesn't complete card entry → checkout expires; no state change

## 3. Live Code Facts & Proposed Changes

### 3.1. Current Webhook Structure

Webhook DTO (`src/webhook/dto/webhook.dto.ts`) already captures:
- `transaction.responseCodeMessage` — failure reason (e.g., "Insufficient Funds")
- `tokenizedCardData.tokenKey` — card token (if `tokenizeCard: true` in request)

**Fact:** Webhook already handles tokenized card data and stores it in `customers.nombaToken` (line 75-80 of webhook.service.ts).

### 3.2. Current Payment Failure Email

Email (`src/webhook/webhook.service.ts` lines 181-213):
- Shows failure reason
- Suggests common causes
- Has "Retry Payment" CTA

**Change Needed:** Add a second CTA or link to the new card update endpoint.

### 3.3. Nomba API Contract

Per Nomba docs (https://developer.nomba.com/docs/api-basics/webhook):
- `POST /v1/checkout/order` creates checkout sessions
- Can be called with `tokenizeCard: true` and **without** an `amount` field
- This generates a card-capture-only URL
- Nomba webhooks with `tokenizedCardData.tokenKey` on success

**Evidence:** Current code already does this in `createSubscriptionPayment` (checkout.service.ts lines 207-219):
```typescript
const orderPayload = {
  order: {
    amount: plan.amount,
    ...
    allowedPaymentMethods: ['Card'],
    subAccountId,
  },
  tokenizeCard: true,
};
```

**For card-update:** We need the same, but with `amount` omitted or set to `0`.

## 4. Implementation Slice

### 4.1. New Endpoint in CheckoutController

**Route:** `POST /api/v1/public/subscriptions/:id/update-payment-method`

**Auth:** None (public)

**Request Body:**
```json
{
  "email": "customer@example.com"
}
```

**Response (200 OK):**
```json
{
  "sessionId": "card_upd_<uuid>",
  "checkoutUrl": "https://checkout.nomba.com/pay/..."
}
```

**Errors:**
- `404 Not Found` — subscription not found
- `409 Conflict` — subscription is canceled
- `502 Bad Gateway` — Nomba checkout generation failed

### 4.2. New Service Method in CheckoutService

**Method:** `updatePaymentMethod(subscriptionId: string, email: string)`

**Steps:**
1. Fetch subscription by ID
2. Validate subscription exists and status is not `canceled`
3. Fetch customer by ID from subscription
4. Validate customer email matches provided email (security check)
5. Call `nombaClient.createCheckoutOrder()` with:
   - `tokenizeCard: true`
   - **No `amount` field** (or `amount: 0`)
   - `subAccountId`
   - Customer email
6. Return checkout URL

### 4.3. Nomba Client Enhancement

**Existing:** `createCheckoutOrder(idempotencyKey, orderPayload)`

**No change needed** — already supports tokenization-only by allowing `amount` to be omitted.

### 4.4. Webhook Handler (No Change Needed)

Existing code already:
- Receives `tokenizedCardData.tokenKey` in `payment_success` webhook
- Stores it in `customers.nombaToken` (lines 75-80)

**For card-update sessions:** Nomba will webhook with a different event type or marker. Need to verify Nomba's behavior.

**Open Question:** Does Nomba send a `payment_success` event for tokenization-only checkouts, or a different event type?

### 4.5. Email Template Update

Update `renderPaymentFailedEmail` in `webhook.service.ts` to include:
- CTA button or link to: `https://lemni.api/api/v1/public/subscriptions/{subscriptionId}/update-payment-method?email={customerEmail}`
- Copy: "Update Payment Method" or "Add a New Card"

## 5. OpenAPI Specification Updates

Add under public checkout endpoints:

```yaml
/api/v1/public/subscriptions/{id}/update-payment-method:
  post:
    summary: Update payment method (card) for a subscription
    description: Generates a tokenization-only checkout URL for the customer to add/update their card details.
    tags:
      - public-api/subscription-management
    parameters:
      - in: path
        name: id
        schema: { type: string }
        required: true
        description: Subscription ID
    requestBody:
      required: true
      content:
        application/json:
          schema:
            type: object
            required: [email]
            properties:
              email:
                type: string
                format: email
                example: customer@example.com
    responses:
      '200':
        description: Card update checkout session created
        content:
          application/json:
            schema:
              type: object
              properties:
                sessionId:
                  type: string
                  example: card_upd_abc123
                checkoutUrl:
                  type: string
                  format: uri
                  example: https://checkout.nomba.com/pay/...
      '404':
        description: Subscription not found
      '409':
        description: Subscription cannot be updated (e.g., already canceled)
      '502':
        description: Payment gateway checkout generation failed
```

## 7. Open Questions

1. **Nomba Event Type for Tokenization-Only Checkouts**
   - When customer completes a tokenization-only checkout (no amount), what event does Nomba send?
   - Is it `payment_success` with `tokenizedCardData`, or a different event type?
   - **Action:** Test in Nomba sandbox or check updated Nomba docs.

2. **Customer Identity Verification**
   - Should we require OTP verification (like unsubscribe) or just email match?
   - **Current Plan:** Email match for MVP (customer already has email from subscription).
   - **Future:** Add OTP for stronger security.

3. **Session Expiry & Duplicate Prevention**
   - If customer opens the update URL twice, do we create two checkout sessions?
   - Should we deduplicate based on `subscriptionId` + `email`?
   - **Current Plan:** Allow multiple sessions; Nomba handles idempotency via our idempotency key logic.
