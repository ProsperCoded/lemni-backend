# RF-004: Customer Self-Unsubscribe via Email Verification (OTP)

**Status:** DRAFT (Awaiting Review)

**Date:** 2026-07-04

**Scope:** Support customer-initiated subscription cancellation using Resend email verification.

---

## 1. Goal
Add public endpoints for customers to unsubscribe from subscriptions without merchant intervention. This will generate a secure 6-digit OTP, send it to the customer using Resend, and confirm the cancellation upon verification.

---

## 2. Requirement Delta

### REQUIREMENTS.md Updates
Add under **4. BILLING & DOMAIN LOGIC (BillingModule)**:
- [ ] **4.6. Customer Self-Unsubscribe**
  - Implement `POST /api/v1/public/subscriptions/:id/unsubscribe/request` to trigger email verification.
  - Implement `POST /api/v1/public/subscriptions/:id/unsubscribe/confirm` to verify OTP and cancel the subscription.
  - Configure Resend integration for transactional emails.

### 3.2. Database Schema / Storage
We can store the active OTPs in Redis since we already have a Redis connection (via BullMQ/ioredis config) or we can create a simple `otp_verifications` table in `schema.ts`.
Let's store them in Redis for fast TTL expiry:
- Key format: `otp:unsubscribe:<subscriptionId>`
- TTL: 10 minutes (600 seconds)

Let's verify if we can access Redis. Yes, we already configure `REDIS_URL`. We can create a lightweight Redis provider in the system or reuse BullMQ connection details to create a standard `ioredis` instance.

### 3.3. Resend Email Client (`src/common/services/email.service.ts`)
