## Summary

Implement a Telegram notification service that sends real-time alerts to merchants about payment events (success, failure, trial expiry, dunning exhaustion, etc.) via their configured Telegram chat ID. Notifications are dispatched asynchronously via BullMQ (`NotificationQueue`) to never block critical billing paths.

---

## Requirement Context

**REQUIREMENTS.md 7.2 (formerly 7.3, renumbered after removing 7.2 Outbound Webhook Dispatcher):**

**Part A: Notification Service (existing)**
```
- [ ] **7.2. Merchant Notification Service (Telegram)**
  - Implement Telegram Bot API integration to send alert notifications to the merchant's configured `telegram_chat_id`.
  - Dispatch notifications asynchronously via a separate BullMQ queue (`NotificationQueue`) so they never block the critical billing path.
  - **Alert Triggers:** Send notifications on key payment events (payment_success, payment_failed, trial_ended, grace_period_exhausted, subscription_canceled, dunning_failed).
  - **Unhappy paths:**
    - Telegram chat ID is invalid or bot is blocked → log the failure; do NOT crash the critical billing path; mark notification as `undelivered`.
    - Telegram API is unavailable → log the notification as `undelivered`; will retry on next queue tick.
    - Notification queue is slow or backed up → notifications may be delayed, but never block transaction processing.
    - Merchant has no `telegram_chat_id` configured → skip notification silently (optional integration).
```

**Part B: Telegram Connection UX (NEW — updates REQUIREMENTS.md 7.2)**

Add to the above requirement:

```
- **Telegram Bot Connection Flow:**
  - Merchant dashboard shows "🤖 Connect Telegram" button (or similar CTA)
  - Button redirects to Telegram deep link: `https://t.me/lemni_bot?start=merchant_<merchantId>`
  - Merchant clicks link → Opens Telegram with bot
  - Merchant types `/start` or Telegram auto-sends it
  - Bot receives `/start merchant_abc123` command
  - Bot verifies merchant ID exists in Lemni database
  - Bot calls Lemni API: `POST /api/v1/admin/telegram/connect` with merchant_id and chat_id
  - Lemni saves chat_id to `merchants.telegram_chat_id`
  - Bot sends confirmation: "✅ Lemni notifications enabled for your account"
  - Merchant sees dashboard update: "✅ Telegram Connected"
  - From now on: Notifications auto-deliver to this chat
  
  - **Unhappy paths:**
    - Merchant ID in deep link is invalid → Bot replies: "Cannot find merchant account. Verify the link is correct."
    - Merchant already connected from different chat → Bot replies: "Telegram already connected. To reconnect, visit your dashboard."
    - Merchant blocks bot → No error; bot simply can't deliver messages (graceful degradation)
    - Merchant clicks "Disconnect" in dashboard → Delete `merchants.telegram_chat_id`, bot no longer receives messages

- **Dashboard Backend Endpoints:**
  - `POST /api/v1/admin/telegram/connect` — Bot calls this to save chat_id after `/start`
    - Auth: Via merchant_id + signature verification (bot signs request with TELEGRAM_BOT_SECRET)
    - Body: `{ merchantId: string, chatId: string }`
    - Response: `{ success: boolean, message: string }`
    - Rate limit: Max 1 connection per merchant per hour (prevent spam)
  
  - `DELETE /api/v1/admin/telegram/disconnect` — Merchant calls to disconnect (JWT auth)
    - Clears `merchants.telegram_chat_id`
    - Bot no longer sends messages to that chat
```

```typescript
  transactionId?: string;
  customerId?: string;
  amount?: number;
  reason?: string; // e.g., card_declined, insufficient_funds, grace_period_exhausted

```

### 3. Service Structure

**File: `src/notification/telegram-bot-handler.service.ts`** — NEW
- `@Injectable()` service to handle bot commands
- Single public method: `handleBotStartCommand(update: TelegramUpdate): Promise<void>`
- Parse `/start merchant_abc123` command from `update.message.text`
- Extract merchant_id from parameter
- Verify merchant exists in DB (return error if not)
- Call `NotificationController.connectTelegram()` endpoint with merchant_id + chat_id
- Send confirmation message back via Telegram API
- Error handling: invalid merchant ID, DB errors, etc.

**File: `src/notification/notification.service.ts`**
- `@Injectable()` service with single public method: `sendTelegramAlert(merchantId: string, message: string): Promise<void>`
- Calls Telegram Bot API to send message to stored chat_id
- Logging for all outcomes (success, skipped, failed)
- Error handling: invalid chat ID (don't retry), API unavailable (log and let BullMQ retry)

**File: `src/notification/telegram-client.ts`**
- Simple wrapper around Telegram Bot API's `sendMessage` endpoint
- Endpoint: `https://api.telegram.org/bot<TOKEN>/sendMessage`
- Payload: `{ chat_id: string, text: string, parse_mode: 'HTML' }`
- Error handling: classify 4xx vs 5xx for retry logic

**File: `src/notification/notification-worker.service.ts`**
- `@Injectable()` service implementing `OnModuleInit` and `OnModuleDestroy`
- Create BullMQ Worker for `NotificationQueue`
- Job handler: format message based on `eventType`, call `NotificationService.sendTelegramAlert()`
- Retry logic: exponential backoff (3 retries: 1 min, 5 min, 30 min) for transient failures
- Final failure: log as `undelivered`, do NOT throw (let job succeed with status: undelivered)

**File: `src/notification/notification.module.ts`**
- `@Module()` with `NotificationService` and `NotificationWorkerService` providers
- Register `NotificationQueue` (BullMQ) with Upstash Redis connection from `SchedulerModule`

**File: `src/notification/dto/notification.dto.ts`**
- Zod schema for NotificationJobPayload
- Export inferred TypeScript type

### 5. New API Endpoints (Telegram Connection)

**File: `src/notification/notification.controller.ts`** — NEW

**Endpoint: `POST /api/v1/admin/telegram/connect`**
- **Auth:** None (bot calls this, uses signature verification instead)
- **Request Body:**
  ```json
  {
    "merchantId": "merchant_abc123",
    "chatId": "123456789",
    "signature": "hmac_sha256_signature_here"
  }
  ```
- **Signature Verification:**
  - Bot signs request: `HMAC-SHA256(merchantId:chatId:timestamp, TELEGRAM_BOT_SECRET)`
  - Backend verifies signature matches
  - Prevents unauthorized chat_id claims
- **Request Validation:**
  - Verify merchant_id exists in database
  - Verify timestamp is recent (within 5 minutes)
  - Rate limit: 1 connection per merchant per hour
- **Response (200 OK):**
  ```json
  { "success": true, "message": "Chat connected successfully" }
  ```
- **Response (400 Bad Request):**
  ```json
  { "success": false, "message": "Invalid merchant or signature" }
  ```
- **Side Effects:**
  - Save `chatId` to `merchants.telegram_chat_id`
  - Log connection event with timestamp
  - Don't overwrite if already connected (unless from same bot/chat_id)

**Endpoint: `DELETE /api/v1/admin/telegram/disconnect`**
- **Auth:** JWT (merchant dashboard)
- **Query Params:** None
- **Side Effects:**
  - Clear `merchants.telegram_chat_id` (set to NULL)
  - Log disconnection event
  - Notifications will no longer be delivered
- **Response (200 OK):**
  ```json
  { "success": true, "message": "Telegram disconnected" }
  ```

**Endpoint: `GET /api/v1/admin/telegram/status`**
- **Auth:** JWT (merchant dashboard)
- **Response (200 OK):**
  ```json
  {
    "connected": true,
    "connectedAt": "2026-07-04T10:15:00Z",
    "chatId": "123456789" (can be masked for display: "1234...6789")
  }
  ```

## Dependencies & Constraints

- **BullMQ:** Already in use for ChargeQueue/DunningQueue; reuse same Redis client
- **Telegram Bot Token:** Set via `TELEGRAM_BOT_TOKEN` env var; if missing or invalid, notifications fail gracefully (logged as `undelivered`)
- **Merchant Telegram Chat ID:** Stored in `merchants.telegram_chat_id` column; optional field (NULL if not configured)
- **No Blocking:** Notification failures never propagate to halt billing operations; they're fully decoupled via async queue
- **Retry Logic:** Use BullMQ's built-in exponential backoff; final failure = logged, not re-enqueued

## Notes

- Notifications are **best-effort**, not guaranteed. If Telegram API is down, merchant doesn't receive alerts.
- No webhook or polling from merchant needed; Lemni pushes alerts to the merchant's Telegram chat.
- This is a **unidirectional** notification service (Lemni → Merchant only), not two-way communication.
- Merchants can disable notifications by:
  - Not setting `telegram_chat_id` in their merchant profile
  - Blocking the bot or leaving the chat
- If `TELEGRAM_BOT_TOKEN` is not set, all notification attempts will fail silently (no crash).
